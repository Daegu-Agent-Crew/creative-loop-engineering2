# CLE3 지식 반영 가이드 — 흥행 분석을 파이프라인에 주입하는 법

> **목적**: "삼체가 왜 인기 있었는가", "역대 흥행 만화는 왜 흥행했는가"에 대한 분석을
> 문서로 보관하는 데 그치지 않고, CLE3 파이프라인이 **실제로 참조·강제·측정**하도록 만드는 설계.
> **동반 문서**: `wiki/principles/three-body-appeal.md`, `wiki/principles/hit-manga-patterns.md`

---

## 핵심 원칙

분석을 "문서로 갖고 있는 것"과 "파이프라인이 실제로 참조하는 것"은 완전히 다르다.
지식이 반영되려면 **판단이 일어나는 지점에 주입**되어야 하며, CLE3 구조상 그 지점은 세 곳이다:

```
[지식화]                [주입]                  [측정]
wiki/principles/  →  prompts/story/v3.md  →  evaluation-rubric.md + PRD 가설(H1~H6)
   원리 문서            출력 필드로 강제          루브릭 항목 + 독자 데이터 검증
                            ↑                        │
                            └────── Phase 7 회고에서 원리 갱신 ──────┘
```

---

## 1단계 — 원리화 (Knowledge Base)

`wiki/principles/` 디렉토리를 신설하고 두 문서를 배치한다.

| 문서 | 내용 |
|---|---|
| `three-body-appeal.md` | 삼체 원작이 사랑받은 이유를 각색 시 따를 수 있는 **실행 규칙**으로 정리 |
| `hit-manga-patterns.md` | 역대 흥행 만화의 **구조 패턴**을 페이지·패널 단위 규칙으로 정리 |

작성 시 철칙: **"삼체는 스케일이 크다" 같은 감상이 아니라, AI가 각색할 때 따를 수 있는 규칙 형태로 쓴다.**

- 나쁜 예: "삼체는 과학적 상상력이 뛰어나다"
- 좋은 예: "추상적 개념(지자, 암흑의 숲)은 반드시 구체적 시각 이미지로 번역한다. 번역 불가능하면 그 개념은 이번 화에서 다루지 않는다"

각 원리에는 ID를 부여한다 (예: `TB-01`, `HM-03`). 이후 단계에서 추적에 사용된다.

---

## 2단계 — 프롬프트 주입 (Phase 1에 연결)

`prompts/story/v3.md`의 [A] 원작 분석·[B] 구조 설계 단계에 다음을 추가한다:

```markdown
## 원리 참조 (필수)
작업 전 wiki/principles/ 의 두 문서를 로드하라.
출력에 다음 필드를 반드시 포함하라:

### applied_principles
| 원리 ID | 이 에피소드에서의 구현 방식 |
|---------|--------------------------|
| TB-xx   | {구체적으로 어떤 장면/구조로 구현했는가} |
| HM-xx   | {구체적으로 어떤 페이지/패널로 구현했는가} |

### cliffhanger_type
이 화의 클리프행거는 hit-manga-patterns의 훅 유형
(미스터리형 / 위기형 / 반전형 / 약속형) 중 무엇인가? 선택 근거 1문장.
```

핵심: 참조를 **선택이 아니라 출력 필드로 강제**한다. "참고하라"고만 쓰면 모델은 건너뛴다.
`schemas/story.schema.json`에 `applied_principles` 배열을 required 필드로 추가하면
기존 스키마 검증 체계가 이 규칙까지 자동으로 지켜준다.

---

## 3단계 — 루브릭 항목화 + 가설 루프 연결 (측정)

### 3-1. 루브릭 추가

원리를 평가 가능한 문장으로 바꿔 `evaluation-rubric.md`에 추가한다:

| Phase | 추가 항목 | 대응 원리 |
|---|---|---|
| 1-[C] | 추상 개념이 구체적 시각 이미지로 번역되었는가 | TB (개념의 시각화) |
| 1-[B] | 떡밥의 심은 화수·회수 예정 화수가 명시되었는가 | HM (떡밥 회계) |
| 1-[D] | 클리프행거 유형이 선언되고 그 유형답게 작동하는가 | HM (훅 설계) |
| 3 | 개인의 감정과 거대 스케일의 대비 컷이 존재하는가 | TB (스케일 대비) |

### 3-2. PRD 가설 루프와 연결

script.md에 이미 존재하는 가설 시스템(H1~H6)이 바로 검증 장치다.
원리가 실제로 독자에게 작동하는지 데이터로 확인한다:

```
원리(가설) → 에피소드에 적용 → 독자 지표 측정(복귀율, Forms 응답)
    ↑                                        │
    └── Phase 7 회고: 지표가 검증한 원리는 유지·강화,
        기각된 원리는 wiki/principles/ 에서 수정 ──┘
```

예: "클리프행거 강도 ↑ → 2화 복귀율 ↑" (H6)이 데이터로 확인되면
HM의 훅 규칙을 유지, 기각되면 원리 문서를 갱신한다.

이 루프가 Phase 7(에피소드 회고)과 맞물리면, 흥행 분석이 정적 문서가 아니라
**매 에피소드마다 더 정교해지는 살아있는 지식**이 된다.
