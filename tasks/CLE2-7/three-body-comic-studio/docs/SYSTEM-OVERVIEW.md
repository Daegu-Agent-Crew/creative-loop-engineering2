# SYSTEM OVERVIEW — Three Body Comic Studio

## 목적
`Three Body Comic Studio`는 삼체 연재 만화 프로젝트를 위한 운영 레이어다. 이 시스템은 작품 제작 자체를 대신하지 않고, 제작 흐름을 구조화하고 품질 루프를 명확히 하며 팀 병렬 작업을 안정적으로 운영하는 역할을 맡는다.

## 시스템 경계

### CLE2가 담당하는 것
- 요구사항 관리
- 에피소드/패널 진행 상태 추적
- 태스크 문서화와 handoff 기준
- 이미지 생성 플레이북과 운영 규칙
- 가설 검증 상태 관리
- PR 리뷰와 Discord 알림의 기준 정의

### `three-body-comic` 저장소가 담당하는 것
- 실제 에피소드 원고
- 콘티, 패널 산출물, 배포용 페이지
- 이미지 생성 스크립트와 결과 아카이브
- 실험 결과 수집 데이터

## 핵심 객체

### Episode
- 단위: 한 화
- 예시: `EP001`, `EP002`
- 책임: 대본, 콘티, 패널 묶음, 배포, 관측의 상위 컨테이너

### Scene
- 단위: 에피소드 내 장면 그룹
- 책임: 패널 묶음의 서사적 구획

### Panel
- 단위: 실제 이미지 생성 및 리뷰의 최소 작업 단위
- 책임: 프롬프트, 시도 횟수, 선택된 결과, 수정 이력 보유

### Hypothesis
- 단위: 파일럿 기간에 검증할 가설
- 우선순위: H1, H2, H5 먼저
- 책임: 관측 항목, 판정 기준, 결과 기록 연결

### Deliverable
- 단위: 팀이 리뷰하는 산출물
- 종류: 콘티 초안, 패널 세트, 에피소드 초안, 실험 리포트

## 정보 흐름
1. 요구사항 또는 실험 목표가 CLE2 이슈/태스크로 등록된다.
2. 에피소드 단위 계획이 `three-body-comic`에서 작성된다.
3. 패널 생성과 리뷰 결과가 운영 문서와 산출물에 기록된다.
4. 배포 후 관측 결과가 `results.md`로 정리된다.
5. 가설 상태를 갱신하고 다음 반복의 입력으로 다시 들어간다.

## 설계 원칙
- 패널 생성 문제와 서사 품질 문제를 분리해 기록한다.
- 자동화가 불완전해도 사람이 개입할 지점을 명시한다.
- 결과물보다 의사결정 근거가 남도록 설계한다.
- 파일럿 5화를 우선 범위로 삼고, 시즌 1 전체 확장은 그 이후 판단한다.

## 후속 구현 포인트
- CLE2에서 Episode/Panel 요약 카드를 별도 뷰로 추가
- `three-body-comic` 저장소에 `episodes/`, `panels/`, `results/` 구조 도입
- GA4 또는 대체 분석 데이터 소스를 results 템플릿에 연결
